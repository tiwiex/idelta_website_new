Section A: Executive Summary

Inspired Delta Ltd submits this proposal for the supply, configuration, hosting, and support of the Addressing Infrastructure Programme Management System (AIPMS). Our recommended delivery model is a hybrid implementation built on ERPNext and the Frappe Framework, with targeted custom modules for the AIPMS Configuration Panel, Field Data Tool (FDT) integration, programme-specific workflows, and reporting requirements.

Proposed Delivery Model

Inspired Delta Ltd proposes a hybrid AIPMS implementation: ERPNext provides the base platform for projects, workflows, documents, procurement, HR, payroll, and reporting; custom Frappe components provide the AIPMS-specific Configuration Panel, payment certification workflows, FDT ingestion, additional dashboards, and field-level governance controls. This is the fastest route to operational readiness while preserving flexibility for future programme changes.

Why ERPNext Enables a Lite Version by Week 2

The credibility of the proposed Lite Version depends on whether the underlying platform already contains the core behaviours required for programme commencement. ERPNext makes the Week 2 Lite target realistic because the first release is not a greenfield software build; it is a controlled configuration of mature modules already capable of handling records, workflows, approvals, documents, notifications, dashboards, and role-based permissions.

For AIPMS, the Lite Version is deliberately focused on Modules 1-5 at core operational level. That means the goal in Weeks 1-2 is not to finish every advanced automation, but to stand up the minimum live controls the programme needs to begin safely: milestone tracking, contract register, claim workflow, document repository, governance records, and baseline RBAC.

ERPNext accelerates this because it already provides the structural building blocks we would otherwise spend weeks writing from scratch: workflow engine, role model, document attachments, print formats, notification rules, list and form views, report framework, and audit-oriented transactional records. Our work in the Lite phase is therefore largely configuration, field design, print/output setup, and approval-path alignment rather than low-level application engineering.

That distinction matters in procurement terms. We are not proposing to use ERPNext as a generic label; we are proposing to use it as a delivery accelerator. The platform already gives us the object model, user administration, workflow states, dashboard widgets, report engine, email alerts, attachment controls, comment trail, and permission system needed to launch an operational first release. Because those foundations already exist, the early project effort can focus on configuring AIPMS-specific behaviour rather than constructing commodity platform features.

In practical terms, this means the Week 2 release is credible because the work sequence is front-loaded around setup activities that ERPNext handles well: defining master data, mapping roles, creating the required DocTypes and fields, configuring workflows, applying print formats, loading reference records, and exposing operational views for programme teams. More complex items such as deep validation rules, advanced payment logic, richer FDT reconciliation, or highly tailored analytics can then be layered in during the subsequent sprints without blocking early programme commencement.

How ERPNext Shortens the Lite Release Path

ERPNext is especially suitable for accelerated government and programme delivery where a controlled operational release is required before every enhancement is complete. Its architecture allows us to work in layers: first establish master data, transactions, approvals, and reporting; then add specialised programme controls once live operations have started. This supports the exact AIPMS requirement of making Modules 1-5 usable early while still progressing toward a more fully tuned production platform.

For AIPMS, the Lite Version will be assembled using a mix of standard ERPNext capability and focused Frappe extensions. Standard capability covers the operating spine of the system: users, roles, lists, forms, workflow states, document attachment, assignments, notifications, activity trails, and structured reports. Focused extension work is then reserved for the areas where the RFP is genuinely programme-specific, such as the Configuration Panel, selective approval paths, certificate generation, controlled import of FDT outputs, and executive reporting views.

This approach is lower risk than a fully bespoke build in the same timeframe. A bespoke application would still be spending the first two weeks creating basic scaffolding such as authentication, form behaviour, approvals, document storage, audit trails, and report structure. ERPNext already has those layers, which is why Inspired Delta Ltd can credibly recommend a Lite Version as a working operational platform rather than a demonstration prototype.

Week 2 Lite Version Positioning

Inspired Delta Ltd positions the Lite Version as a configuration-led operational release, not as a claim that all seven modules will be fully completed within two weeks. The delivery sequence is as follows:

  • Week 1: mobilise, load parameters, create users, define workflows, prepare print formats, and configure the core data model.
  • Week 2: activate Modules 1-5 for live use at core level, validate access and approvals, and begin programme operations.
  • Weeks 3-8: extend the platform into deeper automation, FDT integration, dashboards, HR/payroll completion, and full go-live hardening.

This is exactly the sort of delivery model ERPNext supports well: early operational release through configuration, followed by targeted customisation where programme-specific logic becomes necessary.

Accordingly, the Lite Version will operate as a usable controlled-release environment for programme commencement, not as a reduced demonstration. Users will be able to transact, route approvals, store evidence, review dashboards, and generate core outputs. The distinction between Lite Version and Full Go-Live is therefore not whether the platform is usable, but the depth of automation, integration maturity, and reporting sophistication available at each stage.

The implementation will therefore expose controlled administrative forms for master settings, approval thresholds, unit rates, milestone percentages, escalation triggers, and report parameters, with change history preserved for governance and audit purposes.

HR Module Delivery Approach

The HR module will be implemented using ERPNext HR and Payroll as the foundation, extended where required for programme-specific deployment controls. This allows AIPMS to support employee master records, contract terms, attendance, deployment assignment, payroll processing, statutory deductions, training records, and bulk onboarding from CSV within a single governed environment.

Our approach is to configure payroll structures, salary components, deduction rules, and approval controls early in the build, then complete detailed remuneration logic and parallel validation before production use. Access to salary, bank, and statutory information will be restricted through role-based and field-level permissions, while line managers and deployment supervisors will only see the operational information required for assignment and attendance administration.

The proposal therefore treats the HR module as a controlled operational subsystem, not a standalone payroll tool. It will be integrated into the wider AIPMS governance model so that staff deployment, attendance, payroll readiness, and reporting remain visible to authorised programme administrators without exposing protected personal or financial data unnecessarily.

FDT Integration Delivery Approach

AIPMS does not replace the Field Data Tool. Inspired Delta Ltd will implement the required interface layer so that AIPMS can receive, validate, log, and apply FDT outputs in support of certification, reporting, and programme control workflows. The initial in-scope method is controlled CSV-based ingestion, with optional API-based integration available as a later enhancement if the final FDT specification permits.

Our Phase 1 approach covers schema validation, mandatory-field checks, duplicate detection, import batch logging, row-level error reporting, and reconciliation controls so that invalid or inconsistent records do not silently enter payment or reporting processes. Imported FDT outputs will be retained as source records, with discrepancy flags and exception handling used to protect downstream certification workflows.

This means the AIPMS implementation will provide a governed interface rather than a blind file upload utility. The intent is to make FDT-driven decisions traceable, reviewable, and defensible from an audit perspective, especially where imported field results affect payment certification, milestone evidence, or executive reporting.

Low-Connectivity Operating Model for HR Attendance and Deployment Forms

For the designated HR attendance and deployment use cases, Inspired Delta Ltd proposes a controlled offline-capable browser workflow delivered as a lightweight custom progressive web application (PWA) rather than unrestricted full-platform offline operation. The form will expose only the minimum approved attendance and deployment fields required for field operations, and it will be designed to open quickly, cache its shell locally, and continue accepting entries when connectivity is weak or temporarily unavailable.

In operating terms, the user first opens the lightweight form while online so the latest form definition, reference values, and user permissions are loaded to the device. If connectivity drops, the user can continue to capture attendance or deployment data and save it locally with a time stamp, user identity, and pending-sync status. Once connectivity is restored, the queued transaction is synchronised back to the central AIPMS environment through a controlled Frappe endpoint and subjected to the same server-side validation, permission checks, deduplication checks, and audit logging as any directly online transaction.

This offline approach is intentionally limited to the specific HR forms identified in scope. It is suitable for low-connectivity attendance and deployment administration, and it avoids overstating the proposal as a completely offline ERP environment. The recommended design pattern is consistent with the broader Frappe ecosystem, where mobile-facing products are already delivered through installable progressive web applications rather than through a fully offline ERP desk experience.

Low-Connectivity Step How the Process Will Work
1. Form preparation Authorised HR or deployment users access the designated attendance or deployment form while online. The system loads the latest allowed form structure, required reference values, user permissions, and validation rules into the browser session.
2. Local capture during low connectivity If connectivity becomes weak or unavailable, the user can still enter attendance or deployment data into the lightweight form. The data is saved locally in the browser on the device together with a time stamp, user identity, and sync status marker.
3. Local queue and status tracking Each saved record remains in a pending-sync queue visible to the user or supervisor. This makes it clear which records are still local drafts and which have already been accepted by the central AIPMS platform.
4. Reconnection and submission When the device reconnects, the queued records are submitted to AIPMS through a controlled API or server endpoint. Submission occurs in sequence so each record can be validated and acknowledged individually.
5. Server-side validation AIPMS validates the submitted record against current business rules, required fields, user permissions, duplicate checks, and any linked employee or deployment references. Invalid or conflicting records are rejected into an exception state rather than silently accepted.
6. Audit and final confirmation Accepted records are committed to the live AIPMS database, audit-logged, and marked as synchronised on the user device. Exception records remain visible for correction and re-submission under controlled review.
