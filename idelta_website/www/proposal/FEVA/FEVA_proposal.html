Section A: Executive Summary

Inspired Delta Ltd proposes a hybrid FEVA delivery model built around a Flutter Android field application for enumerators and supervisors, and a Python/Frappe backend for APIs, project management workflows, QA/QC review, reporting, export generation, and geospatial processing. This structure aligns directly with the FEVA RFP: the mobile layer satisfies the Android, offline, SQLite, and device-performance requirements, while the backend provides the structured workflow, auditability, GIS integration support, and data-delivery controls required by IGSL.

The solution is designed for development inside IGSL-owned GitHub repositories and IGSL-managed infrastructure. Inspired Delta Ltd will work within IGSL’s engineering environment, comply with IGSL branching, testing, and code-quality requirements, and deliver all source code, GIS integration scripts, migration logic, and technical documentation into IGSL-controlled repositories.

Field Compliant

Flutter Android app with offline SQLite storage, sync queue, GPS capture, and photo evidence.

Backend Strong

Python/Frappe backend for APIs, dashboards, QA/QC, exports, and supervisory workflows.

GIS Ready

Shapefile ingestion, boundary validation, GeoJSON/Shapefile exports, and live GIS connectivity support.

Delivery commitment: Week 1 design package, Week 3 functional prototype, Week 4 beta, Week 5 pilot in at least 2 LGAs, Week 6 production release, and Week 7 enumerator/supervisor training completion.

Section B: Technical Architecture & Approach

Frontend / Mobile Layer

  • Flutter Android application for enumerators and supervisors.
  • Offline-first local storage using SQLite for records, media queue metadata, and reference layers.
  • GPS capture with accuracy indicator and minimum 5 decimal-place precision handling.
  • Geotagged photo capture with compression and deferred sync.
  • Offline boundary lookup against preloaded State Shapefile-derived layers.

Backend / Platform Layer

  • Python/Frappe backend exposing REST APIs for sync, authentication, workflows, dashboards, and exports.
  • PostgreSQL/PostGIS on IGSL-provisioned infrastructure for live project data and spatial operations.
  • Supervisor review, QA/QC workflows, duplicate checks, audit trail, and project dashboards.
  • GeoJSON and Shapefile export services aligned to WGS 84 / EPSG:4326.
  • Shapefile ingestion and GIS integration scripts committed to IGSL-owned repositories.

Section C: Geospatial Integration, Functional Scope & Delivery Model

Area Approach
Shapefile Ingestion Backend ingestion pipeline imports administrative boundaries, LGA extents, wards, and postcode zones into PostGIS. Incoming datasets are normalized to WGS 84 / EPSG:4326 and repackaged into lightweight mobile boundary assets.
Offline Boundary Use Flutter app ships with preloaded State boundary layers for offline LGA/ward/zone assignment and containment checks at point of capture.
Live GIS Connectivity Backend and mobile integration plan supports WMS, WFS, or documented REST access to the State GIS platform when online, with a configuration layer designed to minimise code changes as the GIS platform matures.
Spatial Validation Street and building GPS points are validated against boundary polygons in real time where possible, and revalidated server-side during sync and QA/QC review. Out-of-bound captures are flagged before approval.
NIPOST Postcode Validation Preloaded postcode boundary layers are checked against GPS capture and reconciled with State administrative boundaries, with discrepancies logged for supervisory review.
Monthly and Final Exports Backend generates GeoJSON, Shapefile, and CSV exports plus photo folder outputs in the structures required by IGSL.

Project Management Features

Operational Controls

  • Role-based authentication for enumerators, supervisors, and administrators.
  • Daily and weekly progress dashboards by enumerator and LGA.
  • Configurable dropdown values managed centrally and pushed without app redeployment where applicable.
  • Helpdesk issue logging and field-support tracking during enumeration.

QA / QC Controls

  • Minimum 5% supervisor sampling workflow.
  • Duplicate GPS proximity checks for building captures.
  • Automated validation for photo presence, GPS precision, street-building linkage, and boundary containment.
  • Coverage and quality reporting for monthly delivery to IGSL.

Development Environment Plan

Inspired Delta Ltd will develop FEVA entirely within IGSL’s GitHub and infrastructure environment. All mobile, backend, spatial, export, and integration code will be committed to IGSL-owned repositories only. The delivery team will comply with IGSL’s branching model, code review controls, CI checks, unit/integration test requirements, dependency scanning, and contributor 2FA requirements.

  • All source code, database migrations, spatial scripts, and build/deploy assets stored in IGSL repositories.
  • No vendor-owned hosting, CI/CD, or DevOps environment used for production delivery.
  • Repository contents include mobile app source, backend source, sync logic, spatial processing scripts, tests, and GIS interface documentation.

Risk Register

Risk Impact Mitigation
Late delivery or revision of State shapefiles Delays to boundary validation and offline packaging Design ingestion pipeline for repeatable updates, begin with provisional datasets, and version all boundary packages.
Unstable field connectivity Delayed uploads and supervision visibility Offline-first mobile architecture with local queueing, retry logic, and sync status indicators.
GPS quality variation by environment Reduced spatial accuracy Use accuracy thresholds, supervisor review flags, and server-side revalidation before approval.
High-volume photo and record sync load Performance bottlenecks during rollout Compress media, separate metadata from asset upload, and stage sync with retry and batching logic.
Field-user adoption issues Slow data capture and quality inconsistencies Run practical training, pilot before full rollout, provide guides, and maintain active helpdesk support.
